# 主流协作文档应用的文档分享与权限系统调研

> 调研日期：2026-03-18
> 调研目标：为 SwarmNote 的去中心化笔记分享系统设计提供参考

---

## 一、飞书/Lark（字节跳动）

### 1. 权限模型

飞书云文档采用四级权限角色体系：

| 角色 | 能力 |
|------|------|
| **所有者（Owner）** | 最高权限，文档创建者，可转让所有权 |
| **可管理（Manager）** | 管理协作者、修改权限设置 |
| **可编辑（Editor）** | 编辑文档内容 |
| **可阅读（Viewer）** | 仅查看文档内容 |

此外，知识库（Wiki）有独立的三级角色：管理员、可编辑成员、可阅读成员。

### 2. 分享机制

- **链接分享**：可设置链接的可访问范围（组织内/互联网所有人）
- **添加协作者**：将用户、群组、用户组或部门添加为协作者并授予权限
- **知识库成员**：通过知识库或共享空间的成员管理批量授权

### 3. 链接分享粒度

- **范围选项**：组织内获得链接的人 / 互联网上获得链接的人
- **权限级别**：链接可授予"可阅读"或"可编辑"权限
- **密码保护**：当链接设置为"互联网可访问"时，可开启加密分享，外部用户需输入密码
- **有效期**：官方文档中未明确提供链接有效期/过期设置功能
- **密级标签**：企业管理员可为文档设置密级（如"内部"），限制对外分享

### 4. 组织结构

```
企业组织
├── 个人空间（我的空间）
├── 共享空间（文件夹树形结构）
│   ├── 文件夹
│   │   ├── 子文件夹
│   │   └── 文档/表格/...
│   └── ...
└── 知识库（Wiki 结构）
    ├── 节点（页面）
    │   ├── 子节点
    │   └── ...
    └── ...
```

### 5. 权限继承

- 文件夹内的文档、子文件夹**默认继承**文件夹的协作者、链接分享范围和权限设置
- 可以单独为文件夹中的内容设置不同的权限（覆盖继承）
- 知识库节点支持"继承父页面树权限"或"自定义子页面树权限"

### 6. 访问控制

- 组织管理员可统一设置对外分享权限、创建知识库权限等
- 支持按部门、用户组批量授权
- 可控制复制/打印/下载/评论权限
- 所有权可转让，离职员工文档自动转移给直属上级

### 7. 特色功能

- **三层权限架构**：企业层 → 知识库/空间层 → 文档/节点层
- **密级管理**：企业级文档分类安全策略
- **部门绑定**：知识库可绑定组织通讯录部门，部门成员自动获得权限
- 组织外匿名用户验证手机号后可编辑文档

## 二、腾讯文档（Tencent Docs）

### 1. 权限模型

腾讯文档采用三级权限：

| 角色 | 能力 |
|------|------|
| **所有者（Owner）** | 最高权限，可转让所有权 |
| **可编辑（Editor）** | 编辑文档、添加评论、更改格式 |
| **可评论（Commenter）** | 查看内容并添加评论，不能修改原文 |
| **可查看（Viewer）** | 仅查看文档内容 |

### 2. 分享机制

- **链接分享**：生成链接发送给他人
- **社交平台分享**：直接分享到 QQ 好友、微信（小程序码形式）
- **指定成员**：从 QQ/微信好友中添加协作成员

### 3. 链接分享粒度

四种链接权限模式：

1. 获得链接的**任何人可查看**
2. 获得链接的**任何人可编辑**
3. **仅指定成员**可查看/编辑
4. **仅自己**可查看/编辑

高级设置：
- **有效期**：支持永久、7天、1天
- **水印**：可为文档添加水印
- **下载限制**：可控制查看者是否能复制/另存/打印

### 4. 组织结构

```
个人空间
├── 文件夹
│   ├── 文档/表格/PPT
│   └── 子文件夹
├── 协作空间（由文件夹升级）
│   └── 多人协作管理
└── 共享文件夹
```

### 5. 权限继承

- 文件夹内文件**默认继承**文件夹权限
- 权限冲突时取角色并集（最高权限优先）
- 协作空间内可统一管理成员权限

### 6. 访问控制

- 企业微信集成时，管理员可对全局和特定组织架构设置权限
- 可限制创建共享空间的人数，设置白名单

### 7. 特色功能

- **表格列级权限**：可对表格的特定列设置保护，限制特定人员的查看/编辑
- **社交化分享**：与 QQ/微信生态深度集成
- **链接有效期**：原生支持有效期设置（永久/7天/1天）
- **水印保护**：文档水印功能

## 三、Notion

### 1. 权限模型

Notion 采用多层次角色体系：

**工作区级别：**

| 角色 | 能力 |
|------|------|
| **Workspace Owner** | 管理工作区设置、成员、安全策略 |
| **Membership Admin**（仅企业版） | 管理成员和权限组 |
| **Member** | 创建/编辑/分享页面 |
| **Guest** | 仅可访问被邀请的特定页面 |

**页面级别：**

| 权限 | 能力 |
|------|------|
| **Full Access** | 编辑内容 + 分享页面 |
| **Can Edit** | 编辑内容，不能分享 |
| **Can Edit Content**（仅数据库） | 创建/编辑数据库页面和属性值 |
| **Can Comment** | 查看 + 评论 |
| **Can View** | 仅查看 |

### 2. 分享机制

- **邀请成员/客人**：通过邮箱邀请特定人员到页面
- **公开链接**：开启"Anyone on the web with link"
- **Teamspace 分享**：通过团队空间批量管理成员访问

### 3. 链接分享粒度

- **公开链接**：任何人（包括非 Notion 用户）可通过链接访问
- **权限级别**：公开链接可设置为可编辑/可评论/仅查看
- **链接过期**：支持设置链接过期时间（可选时间或日期）
- **密码保护**：不支持原生密码保护，需要第三方工具
- **企业版控制**：管理员可全局禁用公开分享

### 4. 组织结构

```
Workspace（工作区）
├── Teamspace（团队空间）- Open / Closed / Private
│   ├── Page（页面）
│   │   ├── Sub-page
│   │   └── Database
│   └── ...
├── Private（个人私有）
│   └── Pages
└── Shared（已分享）
    └── Pages
```

### 5. 权限继承

- **向下级联**：父页面的权限自动应用到所有子页面和数据库
- **最高权限胜出**：当用户通过多种途径获得权限时，取最高权限
- **可覆盖**：可以在子页面单独设置不同权限

### 6. 访问控制

- Teamspace 分三种可见性：Open（任何人可加入）、Closed（可见但需邀请）、Private（仅成员可见）
- Guest 只能逐页邀请，不能获得工作区级访问
- 企业版支持 SAML SSO、域管理
- 可限制成员添加 Guest、移动页面到其他工作区、导出内容

### 7. 特色功能

- **Teamspace 三级可见性**：Open / Closed / Private 精细控制
- **角色组（Groups）**：批量管理权限
- **链接过期时间**：原生支持
- **"最高权限胜出"规则**：多途径权限取最高值
- **数据库专属权限**：Can Edit Content 仅在数据库场景可用

## 四、Google Docs

### 1. 权限模型

**个人文件：**

| 角色 | 能力 |
|------|------|
| **Owner** | 完全控制，包括删除和转让所有权 |
| **Editor** | 编辑内容，默认可修改分享设置 |
| **Commenter** | 评论和建议，不能直接修改 |
| **Viewer** | 仅查看 |

**共享云端硬盘（Shared Drives）额外角色：**

| 角色 | 能力 |
|------|------|
| **Manager** | 添加/移除成员，管理所有内容 |
| **Content Manager** | 添加/编辑/移动/删除文件 |
| **Contributor** | 添加和编辑文件 |
| **Commenter** | 评论 |
| **Viewer** | 查看 |

### 2. 分享机制

- **指定用户**：通过 Google 账号邮箱邀请
- **链接分享**：设置通用访问级别
- **Visitor Sharing**：无 Google 账号的用户通过 PIN 码验证访问（7天有效）

### 3. 链接分享粒度

- **Restricted（受限）**：默认值，仅指定用户可访问
- **组织内**（Workspace 账号）：同域名用户可访问
- **Anyone with the link**：任何人可访问
- 链接可设置为 Viewer / Commenter / Editor
- **过期时间**：支持按用户设置过期日期（仅 Workspace 付费版，最长1年）
- **密码保护**：不支持原生密码保护
- **下载限制**：可禁止 Viewer/Commenter 下载、打印、复制

### 4. 组织结构

```
Google Drive
├── My Drive（我的云端硬盘）
│   ├── Folders
│   │   └── Files
│   └── Files
├── Shared Drives（共享云端硬盘）- 组织级
│   ├── Folders
│   │   └── Files
│   └── Files
└── Shared with me（与我共享）
    └── Files/Folders
```

### 5. 权限继承

- 2025年9月起，Google 强制实施文件夹权限继承
- 文件夹内文件**必须继承**父文件夹权限，不能给予更低权限
- 如需不同权限，必须将文件移到独立文件夹
- Shared Drives 的成员权限适用于其中所有内容

### 6. 访问控制

- 可禁止 Editor 修改权限和分享
- 可禁止 Viewer/Commenter 下载、打印、复制
- Workspace 管理员可限制外部分享
- 支持域级白名单

### 7. 特色功能

- **Visitor Sharing**：非 Google 用户通过 PIN 码验证（7天有效期）
- **强制权限继承**：2025年起文件夹权限继承成为强制行为
- **Shared Drives 五级角色**：比个人文件更精细的角色体系
- **Workspace 管理员控制**：企业级安全策略
- **建议编辑（Suggesting Mode）**：Commenter 可提出编辑建议

## 五、语雀（Yuque，阿里巴巴）

### 1. 权限模型

语雀采用三层架构对应的权限体系：

**空间/团队级别：**

| 角色 | 能力 |
|------|------|
| **空间管理员** | 管理整个空间的设置和成员 |
| **团队管理员** | 管理团队及其下属知识库 |
| **成员** | 根据所在团队/知识库的角色行使权限 |

**知识库级别：**

| 角色 | 能力 |
|------|------|
| **可管理** | 管理知识库设置、协作者 |
| **可编辑** | 编辑知识库中的文档 |
| **可阅读** | 仅查看知识库内容 |

### 2. 分享机制

- **知识库协作者**：搜索用户名添加，被邀请者收到消息+邮件通知
- **团队成员**：将人员添加到团队，自动获得团队下所有知识库权限
- **链接分享**：开启后无需登录即可查看
- **部门绑定**：知识库可绑定组织通讯录部门

### 3. 链接分享粒度

- **可见范围**：仅协作者 / 空间所有成员 / 互联网所有人
- **密码访问**：支持为分享链接设置密码，支持重置密码
- **有效期**：支持设置链接有效期
- **关闭分享**：可随时关闭分享链接
- **空间限制**：空间内容不能设置互联网公开（需通过分享功能单独开启）

### 4. 组织结构

```
企业空间
├── 团队
│   ├── 知识库 A
│   │   ├── 文档
│   │   └── 文档
│   └── 知识库 B
├── 个人
│   └── 知识库
└── ...
```

三层结构：**文档 → 知识库 → 团队/空间**

### 5. 权限继承

- 团队成员自动拥有团队下**所有知识库**的权限
- 知识库可见范围必须**从属于**团队可见范围
- 知识库协作者仅拥有特定知识库的权限，不等同于团队成员

### 6. 访问控制

- "仅空间成员可访问"作为安全约束
- 支持操作日志、对外分享记录、访问记录审计
- 可批量管理团队/知识库成员权限

### 7. 特色功能

- **知识库协作者 vs 团队成员**：两种维度的权限授予方式
- **部门绑定**：与组织通讯录深度集成
- **加密文本**：文档内可插入需密码才能查看的加密文本段落
- **安全审计**：完整的操作日志和分享记录
- **链接密码 + 有效期**：原生支持

## 六、Obsidian（Publish & Sync）

### 1. 权限模型

**Obsidian Sync（同步）：**

Obsidian Sync 的权限模型**极为简单**，几乎没有细粒度控制：

| 角色 | 能力 |
|------|------|
| **Vault Owner** | 完全控制，包括邀请/移除协作者 |
| **Collaborator** | 与 Owner 几乎相同的权限（除了用户管理） |

所有协作者对 Vault 中的所有文件拥有**完全相同**的读写权限。

**Obsidian Publish（发布）：**

- 只有 Vault Owner 一个角色
- 发布是单向的：Owner 选择哪些笔记发布到网站

### 2. 分享机制

- **Sync**：邀请用户连接到同一个远程 Vault，双向同步
- **Publish**：将笔记发布为静态网站
- 无内置的文档级分享功能

### 3. 链接分享粒度

- **Publish**：发布后为公开网站链接
- **密码保护**：仅支持**全站密码**，不支持单页/文件夹级别的密码
- **过期时间**：不支持
- **访问控制**：无用户级别的访问控制，无页面/文件夹级权限

### 4. 组织结构

```
Vault（仓库）
├── Folder
│   ├── Note.md
│   └── Sub-folder
└── Note.md
```

纯文件系统结构，无工作区/团队/空间等组织层级。

### 5. 权限继承

- **Sync**：无权限继承概念，所有协作者对所有文件有完全权限
- **Publish**：无权限继承，密码保护是全站级别

### 6. 访问控制

- Sync：移除用户后立即失去远程 Vault 访问权，本地副本保留
- Publish：全站密码或完全公开，二选一
- 配置同步开启时，任何协作者的配置修改会影响所有用户

### 7. 特色功能

- **本地优先（Local-first）**：数据始终保存在本地，Sync 是可选的
- **纯文件系统**：Markdown 文件，无需特殊格式
- **极简权限**：刻意保持简单，不追求企业级权限管理
- 社区强烈要求但**至今未实现**的功能：页面级密码保护、用户级认证、基于角色的访问控制

## 八、对 SwarmNote 的启示

### 1. P2P 去中心化场景的权限挑战

上述产品均依赖中心化服务器来管理和执行权限。SwarmNote 作为去中心化 P2P 应用，权限执行需要不同的机制：

- **密码学方式**：使用加密而非服务器端访问控制来实现权限
- **签名验证**：使用公钥/签名来验证编辑权限
- **加密分级**：只读权限 = 给予解密密钥；编辑权限 = 给予签名密钥或 CRDT 写入权限

### 2. 可参考的设计要素

| 要素 | 建议参考 | 理由 |
|------|---------|------|
| 权限角色 | 腾讯文档/Google Docs 的 3-4 级 | 简单直观，适合个人/小团队 |
| 分享机制 | 链接分享 + 密码保护 | 无需中心化用户系统 |
| 有效期 | 腾讯文档的简单有效期 | 通过加密密钥轮换实现 |
| 组织结构 | Obsidian 的 Vault/Folder 结构 | 本地优先，文件系统友好 |
| 权限继承 | Notion 的向下级联 + 可覆盖 | 灵活但不复杂 |

### 3. SwarmNote 可能的权限模型草案

```
文档权限级别:
- Owner（所有者）: 完全控制，持有主密钥
- Editor（编辑者）: 可读写，持有写入密钥
- Viewer（查看者）: 只读，持有只读解密密钥

分享方式:
- 设备间同步: 同一身份的多设备自动同步（P2P）
- 邀请协作: 通过公钥交换 + 配对码
- 链接分享: 加密链接，内嵌解密密钥（类似 Mega.nz 的模式）

权限执行:
- 写入权限通过 CRDT 操作签名验证
- 只读权限通过对称加密密钥分发
- 权限撤销通过密钥轮换 + 重新加密
```
